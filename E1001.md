# E1001 本地图片显示方案

本文档把 `E1002.md` 中的本地图片显示方案迁移到 Seeed Studio reTerminal E1001。迁移后的总体方案仍然是：Linux 主机通过局域网 HTTP 请求控制 E1001，E1001 从 MicroSD 卡读取 BMP 图片并刷新到 ePaper 屏幕。区别在于 E1001 是 7.5 寸 800x480 黑白 ePaper，不是 E1002 的 7.3 寸全彩 Spectra 6 ePaper。

官方参考：

- E1001 入门 Wiki：`https://wiki.seeedstudio.com/getting_started_with_reterminal_e1001/`
- Seeed ePaper Arduino 说明：`https://wiki.seeedstudio.com/epaper_work_with_arduino/`
- reTerminal E 系列 Arduino Cookbook：`https://wiki.seeedstudio.com/reterminal_e10xx_with_arduino/`

## 迁移结论

E1002 方案可以迁移到 E1001，但必须更新以下内容：

| 项目 | E1002 | E1001 |
| --- | --- | --- |
| 屏幕 | 7.3 寸全彩 ePaper | 7.5 寸黑白 ePaper |
| 分辨率 | 800x480 | 800x480 |
| 颜色处理 | 建议使用黑、白、红、绿、蓝、黄 | 必须按黑白或灰度抖动处理 |
| 屏幕控制器配置 | `BOARD_SCREEN_COMBO 521` | `BOARD_SCREEN_COMBO 520` |
| `driver.h` 注释 | `reTerminal E1002 (UC8179C)` | `reTerminal E1001 (UC8179)` |
| Arduino 示例入口 | Colorful ePaper 示例 | Basic / Monochrome ePaper 示例 |
| SD 卡 | FAT32，最大 32 GB | FAT32，最大 32 GB |
| Wi-Fi | 2.4 GHz | 2.4 GHz |

## 整体架构

```text
Linux 主机
  |
  | 局域网 Wi-Fi HTTP 请求
  v
E1001 ESP32-S3 HTTP 服务
  |
  | 从 MicroSD 卡读取 BMP 图片
  v
800x480 黑白 ePaper 屏幕
```

这个方案不依赖 Home Assistant、SenseCraft HMI 或云平台。注意：烧录自定义 Arduino 固件后，设备会运行本地 HTTP 服务方案，不再使用原厂 SenseCraft HMI 工作流；如果要恢复原厂体验，需要按官方 Wiki 重新刷回对应固件。

## 验证标准

完成迁移后，需要满足以下条件：

1. Arduino 串口监视器输出 E1001 已连接 Wi-Fi，并显示局域网 IP。
2. 浏览器或 `curl http://<E1001_IP>/status` 返回正常状态。
3. `curl http://<E1001_IP>/list` 能列出 SD 卡根目录里的 BMP 文件。
4. `curl http://<E1001_IP>/show?img=menu` 后，E1001 能显示 `menu.bmp`。
5. 图片显示为清晰黑白或黑白抖动效果，没有沿用 E1002 的六色设计。

## 第一步：确认 E1001 官方基础要求

E1001 官方规格中和本方案相关的部分如下：

- 处理器：ESP32-S3，8 MB PSRAM。
- 显示屏：7.5 寸 Monochrome ePaper。
- 分辨率：800x480。
- 存储：32 MB Flash，支持 MicroSD 卡。
- MicroSD：最大支持 32 GB，FAT32 格式。
- 无线：2.4 GHz 802.11 b/g/n Wi-Fi，Bluetooth 5.0。
- 供电：USB-C 5V/1A，也可使用内置电池。

建议首次拿到设备后先按官方 Wiki 完成一次固件更新。官方提示标准 Flash 会保留 Wi-Fi、已部署页面和图片；Full Flash 会清空所有数据并恢复到类似出厂状态。做本地自定义固件开发时，如果设备睡眠导致电脑识别不到刷写进度，先按顶部绿色 Wake / Refresh 按钮唤醒设备再重试。

## 第二步：准备 MicroSD 卡

E1001 只支持最大 32 GB、FAT32 格式的 MicroSD 卡。很多 32 GB 卡默认是 exFAT，需要重新格式化为 FAT32。

Linux 上可以先查看设备名：

```bash
lsblk
```

假设 SD 卡设备是 `/dev/sdX1`，格式化命令如下。执行前必须确认设备名，避免格式化错盘。

```bash
sudo mkfs.vfat -F 32 /dev/sdX1
```

建议 SD 卡根目录保持简单：

```text
SD 卡根目录/
├── menu.bmp
├── info.bmp
├── promo.bmp
└── idle.bmp
```

## 第三步：准备 E1001 黑白 BMP 图片

E1001 和 E1002 的分辨率相同，仍然使用 **800x480**。但 E1001 是黑白屏，因此不要沿用 E1002 的六色素材。建议把图片处理为黑白或黑白抖动效果，再保存为 24-bit BMP，便于复用 E1002 本地读取 BMP 的固件逻辑。

安装 ImageMagick：

```bash
sudo apt install imagemagick
```

适合图标、菜单、文字界面的硬阈值转换：

```bash
convert 原始图片.png \
  -resize 800x480! \
  -colorspace Gray \
  -threshold 55% \
  -type TrueColor \
  -define bmp:subtype=RGB \
  BMP3:menu.bmp
```

适合照片或渐变图的黑白抖动转换：

```bash
convert 原始图片.png \
  -resize 800x480! \
  -colorspace Gray \
  -ordered-dither o8x8,2 \
  -type TrueColor \
  -define bmp:subtype=RGB \
  BMP3:photo.bmp
```

批量转换 PNG 文件：

```bash
mkdir -p bmp_out

for f in *.png; do
  convert "$f" \
    -resize 800x480! \
    -colorspace Gray \
    -ordered-dither o8x8,2 \
    -type TrueColor \
    -define bmp:subtype=RGB \
    BMP3:"bmp_out/${f%.png}.bmp"
done
```

把输出的 `.bmp` 文件复制到 MicroSD 卡根目录，然后插入 E1001。

## 第四步：搭建 Arduino 开发环境

在 Arduino IDE 中打开 **File -> Preferences**，添加 ESP32 board URL：

```text
https://raw.githubusercontent.com/espressif/arduino-esp32/gh-pages/package_esp32_index.json
```

然后在 Boards Manager 中安装 **esp32 by Espressif Systems**。板型选择：

```text
XIAO_ESP32S3
```

安装 Seeed_GFX 库：

1. 从 `https://github.com/Seeed-Studio/Seeed_GFX` 下载 ZIP。
2. 在 Arduino IDE 中选择 **Sketch -> Include Library -> Add .ZIP Library**。

如果本机已经安装过 `TFT_eSPI`，官方 Arduino 文档提示可能需要临时移除或重命名该库，避免和 `Seeed_GFX` 冲突。

**Tools 菜单必须启用 OPI PSRAM**：

```text
Tools -> PSRAM -> OPI PSRAM
```

## 第五步：新建 E1001 的 `driver.h`

在 Arduino sketch 文件夹同级新建 `driver.h`。E1001 对应配置必须改为：

```cpp
#define BOARD_SCREEN_COMBO 520  // reTerminal E1001 (UC8179)
```

这是迁移中最关键的差异。E1002 使用的是：

```cpp
#define BOARD_SCREEN_COMBO 521  // reTerminal E1002 (UC8179C)
```

如果这里仍然保留 `521`，E1001 很可能黑屏、不刷新，或者显示异常。

## 第六步：调整并烧录 Arduino 固件

在 E1002 方案的 `e1002_server.ino` 基础上迁移时，建议只做必要改动：

1. 将 sketch 另存为 `e1001_server.ino`，避免混淆设备型号。
2. Wi-Fi SSID 和密码改成现场 2.4 GHz Wi-Fi。
3. 确认 `driver.h` 和 `e1001_server.ino` 在同一个文件夹。
4. 确认 `driver.h` 使用 `BOARD_SCREEN_COMBO 520`。
5. 确认所有图片处理逻辑按黑白屏幕设计，不再使用 E1002 的六色 palette 假设。
6. Arduino IDE 选择 `XIAO_ESP32S3`。
7. 设置 **Tools -> PSRAM -> OPI PSRAM**。
8. 通过 USB-C 连接 E1001，设备背面电源开关拨到 ON。
9. 上传固件。
10. 打开串口监视器，波特率设置为 115200，等待输出 IP 地址。

串口输出示例：

```text
WiFi connected!
IP Address: 192.168.1.88
```

如果电脑能看到串口但上传一直没有进度，先按顶部绿色 Wake / Refresh 按钮唤醒设备。必要时，按 ESP32-S3 常规方式进入下载模式后再上传。

## 第七步：Linux 端发送 HTTP 指令

### 使用 `curl`

把 IP 替换成串口输出的 E1001 实际地址：

```bash
# 切换图片，固件会从 SD 卡读取 /menu.bmp
curl "http://192.168.1.88/show?img=menu"

# 查询当前状态
curl "http://192.168.1.88/status"

# 列出 SD 卡上的所有图片
curl "http://192.168.1.88/list"
```

### 使用 Python 脚本

如果沿用 E1002 方案中的 `e1002_controller.py`，建议复制为 `e1001_controller.py`，只修改类名、提示文本和默认说明，不改变 HTTP API：

```bash
python3 e1001_controller.py --ip 192.168.1.88 --list
python3 e1001_controller.py --ip 192.168.1.88 menu
python3 e1001_controller.py --ip 192.168.1.88 info --wait
```

如果暂时不重命名脚本，也可以继续使用原脚本，只要 IP 和接口路径一致：

```bash
python3 e1002_controller.py --ip 192.168.1.88 --list
python3 e1002_controller.py --ip 192.168.1.88 menu
```

### 在自己的 Python 程序中调用

```python
from e1001_controller import E1001Controller

screen = E1001Controller("192.168.1.88")
screen.show("menu")
screen.show("idle")
```

## 第八步：固定 IP 和现场部署

建议在路由器中给 E1001 的 MAC 地址绑定固定 IP。这样 Linux 主机端脚本可以长期使用同一个地址，不需要每次重启后查看串口日志。

部署前检查：

1. E1001 和 Linux 主机在同一个局域网。
2. Wi-Fi 是 2.4 GHz，不是 5 GHz。
3. 路由器没有开启客户端隔离。
4. MicroSD 卡为 FAT32，容量不超过 32 GB。
5. SD 卡根目录图片文件名和 HTTP 参数一致，例如 `show?img=menu` 对应 `menu.bmp`。

## 关键注意事项

**关于刷新时间：** ePaper 不是 LCD。E1001 黑白 ePaper 刷新仍然需要等待，不适合高频动态画面。HTTP `/show` 接口可以先返回，实际刷屏在设备端继续进行。上层程序应避免连续快速发送大量切换请求。

**关于颜色：** E1001 是黑白屏。所有 UI 设计都应按黑白高对比思路重新处理，不要直接复用 E1002 的红、绿、蓝、黄强调色。需要层次时使用线条粗细、图标形状、排版留白和黑白抖动，而不是依赖颜色。

**关于睡眠：** 官方 Wiki 提到设备无操作后会进入睡眠以省电。使用自定义固件时，如果你保留了低功耗逻辑，需要保证 HTTP 服务、Wi-Fi 和刷新任务的唤醒策略是明确的。调试阶段建议先禁用深度睡眠，确认显示和 HTTP 控制稳定后再考虑省电策略。

**关于 SenseCraft：** 官方入门方案推荐 SenseCraft HMI，用于无代码设计和云端部署。本地 HTTP + SD 卡方案是自定义固件路线，两者不是同一套运行路径。烧录自定义固件后，本地方案优先；要回到 SenseCraft，需要重新刷官方固件并重新配网/绑定。

**关于代码命名：** HTTP API 可以继续保持 `/show`、`/status`、`/list` 不变，这样 Linux 端调用成本最低。但文件名、类名和文档建议改成 E1001，避免后续维护时把 `BOARD_SCREEN_COMBO 520` 和 `521` 混用。
