# 食光破壁者：现代化餐厅无障碍协同点餐系统
### Product & Technical Document · Hackathon Edition

---

## 一、产品经理评审总结

### 🏅 创新性评分：9/10

**核心洞察极强。** 市面上绝大多数"适老化"产品的逻辑是错的——它们默认用户需要被"教育"来适应技术。本项目的价值观反转了这个命题：

> **"不是让人去适应机器，而是让机器主动弯下腰来服务人。"**

这句话本身就是完整的产品哲学，也是评委席上最容易引发共鸣的那种陈述。健康禁忌主动推荐（高血压自动屏蔽高盐菜品）是本项目最具差异化的功能点，这不是"便利"，而是真正意义上的健康赋能。

---

### 🎯 抓人眼球程度：9/10

三个关键时刻构成了完整的情感弧线：

| 时刻 | 情感锚点 |
|------|---------|
| 机器人主动驶来、机械臂降低高度 | 「被看见」的尊严感 |
| 语音对话 + 菜单主动筛选 | 「被照顾」的安全感 |
| 生日舞蹈 + 现场合影 | 「被珍视」的温暖感 |

这三个层次层层递进，前两个解决功能痛点，第三个制造记忆点。评委在30秒内不会忘记一台机械臂跳舞的画面。

---

### ⚙️ 技术可行性评分（24小时内）：8/10

| 模块 | 可行性 | 风险点 |
|------|--------|--------|
| 手机H5控制界面 | ✅ 高 | 无 |
| 机械臂高度预设轨迹 | ✅ 高 | 调试时间需预留 |
| 生日舞蹈预录动作 | ✅ 高 | 节拍同步需手动对齐 |

## 二、关于新增功能的取舍建议

### ❌ 建议：砍掉"人坐上底盘被送到餐桌"的现场实现

这个想法在**叙事层面极好**，但在**24小时黑客松的现场实现层面，强烈建议不做**。理由如下：

**安全风险是第一红线。**
让真人坐在移动底盘上，哪怕只是一位模拟"老人"的队友，在现场嘈杂环境下出现任何碰撞、急停或侧翻，轻则演示失败，重则人员受伤。任何评委看到这个场景，第一反应都是担忧，而不是感动。一旦出现安全事故，黑客松直接终止，前面所有的工作归零。

**工程复杂度无法消化。**
底盘承重、低速稳定控制、紧急停止逻辑——这三件事单独拿出来每一件都需要专项调试时间，24小时内同时保证它们可靠运行的概率极低。更糟的是：这个模块出问题，会连带拖垮整个Demo的节奏。

**正确做法：把它放进"未来规划"PPT，用一句话说出来。**
这个想法的概念价值非常高，完全可以作为路演时的"产品愿景"展示：

> "未来，当老人腿脚不便、无法走到餐桌时，底盘本身也可以变成一辆静音电动轮椅，把老人从门口温柔地送到座位前。今天的Demo是第一步，而整个旅程，才刚刚开始。"

**这样说，比实际做更安全、更有力量。**

## 三、产品文档

### 3.1 产品定位

**产品名称：** 食光破壁者（Barrier Breaker）
**赛道：** 无障碍科技 / 具身智能应用
**目标用户：** 65岁以上老年人、轮椅用户、视障人士

**核心价值主张：**
在每一家普通餐厅，无需改造任何固定设施，只需一套可移动的协同系统，就能让无障碍就餐成为现实——不是靠用户的努力，而是靠系统的主动适配。

---

### 3.2 解决的核心痛点

当下餐厅数字化改造存在一个系统性盲区：

- 扫码点单默认用户能看清屏幕、能操作智能手机；
- 高台点餐机默认用户能站立或抬手；
- 营养信息默认用户能自行辨别忌口风险。

这三个"默认"，对视障人士、轮椅用户和有慢性病的老年人是三道隐形的墙。

---

### 3.3 功能规格

#### 功能一：主动靠近与自适应高度

**触发方式：** 服务员或用户通过手机H5界面点击"呼叫服务"

**执行流程：**
1. 服务员遥控底盘平稳驶向目标餐桌
2. RDK X5摄像头检测用户上半身/轮椅高度
3. 机械臂执行预设IK轨迹，将屏幕从标准高度降低至用户平视角度（轮椅适配高度约0.6m）
4. 屏幕自动旋转至最佳观看角度（约15°前倾）

**硬件依赖：** 移动底盘、6轴机械臂、RDK X5、电子屏幕

---

#### 功能二：多模态无障碍点餐

**触发方式：** 机械臂到位后自动启动

**执行流程：**
1. 屏幕切换至"长辈模式"：字体≥24px、高对比度（黑底白字）、两列极简布局
2. 语音模块（手机H5的Web Speech API）主动问询："您好，请问有饮食忌口吗？"
3. 用户语音回复（如"我有高血压"/"不能吃甜的"）
4. 语音转文字后调用DeepSeek/GPT API进行语义解析
5. 系统自动：将高盐/高糖/高油菜品置灰或隐藏，高亮推荐符合健康条件的菜品，语音播报推荐结果
6. 用户语音或触屏确认点单，指令通过WebSocket传至后端，同步推送到收银系统

**技术亮点：** 健康禁忌主动推荐是差异化核心，同时解决了"老年人慢性病饮食难选择"和"视障人士无法看图点餐"两个痛点。

---

#### 功能三：生日快乐模式（现场高潮）

**触发方式：** 服务员或顾客在屏幕上点击"生日模式"

**执行流程：**

| 步骤 | 手机端行为 | 机器人端行为 |
|------|-----------|------------|
| 1 | 播放《生日快乐歌》（HTML5 Audio本地文件）| 接收 `{"action": "start_birthday_dance"}` 指令 |
| 2 | 全屏显示祝福动画 | 执行预录关节轨迹：左右摇摆→画心形→节奏点头，配合底盘轻微原地旋转 |
| 3 | 屏幕显示"3、2、1——茄子！"倒计时 | 机械臂将摄像头转向顾客 |
| 4 | 触发摄像头拍照 | — |
| 5 | 照片实时显示在屏幕上，生成二维码供扫码下载 | — |

### 4.3 关键技术实现细节

#### 机械臂高度自适应

```python
# 伪代码：根据检测到的人脸高度计算目标关节角度
def auto_lower_arm(detected_face_y_ratio):
    # detected_face_y_ratio: 0.0 = 画面顶部, 1.0 = 画面底部
    # 正常站立用户约0.3，轮椅用户约0.6
    
    if detected_face_y_ratio > 0.5:  # 轮椅/坐姿
        target_preset = "wheelchair_mode"  # 预设轨迹：降至0.6m
    else:
        target_preset = "seated_mode"      # 预设轨迹：降至0.9m
    
    arm_controller.execute_preset(target_preset)
```

#### 健康推荐调用（DeepSeek API）

```python
def filter_menu_by_health(user_speech: str, menu_items: list) -> dict:
```

#### 生日舞蹈同步

```python
# 预录关节角度序列（已通过示教器录制）
BIRTHDAY_DANCE_SEQUENCE = [
    {"joints": [0, 30, -20, 0, 45, 0],  "duration": 0.5},  # 左摆
    {"joints": [0, -30, -20, 0, -45, 0], "duration": 0.5}, # 右摆
    {"joints": [0, 0, 30, 60, 0, 0],    "duration": 0.8},  # 画圆弧
    # ... 共约16个关键帧，总时长约30秒
]

def start_birthday_dance():
    # 手机端同时触发音乐播放
    for keyframe in BIRTHDAY_DANCE_SEQUENCE:
        arm.move_to_joints(keyframe["joints"])
        time.sleep(keyframe["duration"])
```

## 五、答辩脚本建议

### 开场白（30秒）
> "2026年，中国65岁以上老年人已超过2.5亿。走进任何一家餐厅，迎接他们的，是一张张需要扫码、点击、放大的二维码菜单。这张二维码，对于腿脚不便的老人、视力衰退的长辈、坐在轮椅上的朋友，是一道看不见的墙。
>
> 我们不想教他们如何翻墙。我们让墙，消失。"

### Demo串联词
> "现在，我们的机器人收到了呼叫信号。它驶向这位老人，弯下腰，把菜单递到他最舒服的视线高度。
>
> '您好，请问有什么忌口吗？'
>
> '我有高血压。'
>
> 菜单变了。高盐、高油的菜品悄悄隐去，清淡的推荐浮现出来，一道声音告诉他：'为您推荐今日清淡套餐。'
>
> 今天，恰好是他的生日。"

### 未来愿景（30秒，加入被砍功能作为展望）
> "这是我们今天能做到的。而我们相信，当底盘本身可以承载一个人的重量，当它能从门口把一位腿脚不便的老人静静地送到餐桌前——那个时候，'无障碍'这三个字，才真正完整。"

## 六、风险与缓解预案

| 风险 | 概率 | 缓解方案 |
|------|------|---------|
| 现场Wi-Fi不稳定，LLM API超时 | 中 | 预备本地规则引擎：关键词→菜品标签的简单映射，离线可用 |
| 机械臂运动抖动 | 低 | 提前调好运动速度参数（speed_ratio降至0.3），稳定优先于帅气 |
| 语音识别在嘈杂环境下失效 | 中 | 备用方案：屏幕上显示「我有高血压」「我需要低糖」等快捷按钮 |
| 生日歌手机音量不够 | 低 | 连接蓝牙音箱，或用笔记本播放音乐 |
| 合影图片无法显示 | 低 | 退化方案：仅显示倒计时动画，合影环节跳过不影响主流程 |
